Hooks run automatically when things happen. Edit a file? Auto-format. About to delete something dangerous? Block it. Session finished? Send a notification.

Unlike instructions in `CLAUDE.md` -- which the agent may or may not follow -- hooks are deterministic. They always fire when triggered.

### How hooks work

A hook is a shell command tied to an event in Claude Code's lifecycle. The events are straightforward: `PreToolUse` fires before a tool call, `PostToolUse` fires after, `Stop` fires when Claude finishes responding, and `Notification` fires when it needs your attention.

Each hook has a `matcher` -- a regex that determines which tool triggers it. A `PostToolUse` hook with matcher `Edit|Write` fires only when Claude edits or writes a file, not when it reads or runs a command.

### Auto-formatting: the most common hook

You probably already run Prettier or a formatter. Why not run it automatically every time Claude touches a file?

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"
          }
        ]
      }
    ]
  }
}
```

Add this to `.claude/settings.json`. Now every file Claude edits gets formatted immediately, without Claude spending tokens on formatting decisions.

### Blocking dangerous actions

`PreToolUse` hooks can block actions entirely. Exit code `2` means "stop -- don't do this." The agent receives your error message as feedback.

Want to prevent Claude from touching `.env`, `package-lock.json`, or anything in `.git/`? Write a small bash script that checks the file path and exits with `2` if it matches a protected pattern. Claude gets the feedback and adjusts its approach.

Exit code `0` means proceed. Any other exit code means proceed but log the error.

### Where hooks live

Hooks are configured in settings files, not memory files. Three locations:

- `~/.claude/settings.json` -- applies to all your projects
- `.claude/settings.json` -- single project, committable to version control
- `.claude/settings.local.json` -- single project, gitignored

Use `/hooks` in Claude Code to browse what's configured.

### Beyond shell commands

Hooks also support `type: "prompt"` (a single-turn LLM evaluation), `type: "agent"` (multi-turn verification), and `type: "http"` (POST to an endpoint). The agent hook type is powerful -- imagine a mini code review that runs automatically after every edit.

### Headless mode

Everything we've discussed works interactively. But Claude Code also runs without a human with `claude -p`:

```bash
claude -p "Run tests and fix any failures" \
  --allowedTools "Bash,Read,Edit"
```

This unlocks CI pipelines, automated migrations, and scheduled maintenance. Use `--allowedTools` to pre-approve specific tools. Use `--output-format json` for structured output. Use `--bare` for minimal startup that skips hooks, MCP, and skills.

Headless mode combined with hooks gives you a fully automated coding pipeline -- the agent writes code, hooks enforce formatting and safety, and CI validates the result.
