Without configuration, the agent re-learns your preferences every session. It guesses your package manager, your test command, your file structure. Sometimes it guesses right. Often it doesn't.

One configuration file changes everything.

### What changes when you add a memory file

Imagine telling a new contractor your project conventions once, and they remember them every day. That's what `CLAUDE.md`, `AGENTS.md`, and `copilot-instructions.md` do for your AI tools. Build commands, code style, architecture decisions, naming conventions -- all loaded before the first prompt.

The agent stops guessing and starts working within your constraints.

### CLAUDE.md (Claude Code)

This is Claude Code's memory file. It loads at session start and walks up the directory tree -- a `CLAUDE.md` in `packages/api/` adds to, not replaces, the one at the project root.

Run `/init` to generate a starting point. Claude analyzes your codebase and drafts build commands, test instructions, and conventions. Refine it with things Claude can't discover: why you chose Prisma over Drizzle, which directories are sensitive, what your team expects in a PR description.

```markdown
# CLAUDE.md
- Use pnpm, not npm
- Run `pnpm test -- --run` for single test files
- API handlers follow src/api/handlers/ pattern
- Always validate inputs with zod schemas
```

For larger projects, break rules into `.claude/rules/` files. These can be scoped to specific file patterns -- a rule in `api-design.md` with `paths: ["src/api/**/*.ts"]` only loads when Claude works on API files.

### AGENTS.md (OpenCode)

OpenCode reads `AGENTS.md` from the project root. Run `/init` to generate one. It works the same way -- persistent instructions that survive across sessions.

OpenCode also falls back to `CLAUDE.md` if no `AGENTS.md` exists. So one file can serve both tools. For additional instruction sources, list them in `opencode.json`:

```json
{
  "instructions": ["CONTRIBUTING.md", "docs/guidelines.md"]
}
```

### copilot-instructions.md (GitHub Copilot)

Place project instructions in `.github/copilot-instructions.md`. These apply to all Copilot Chat interactions in the repository. Copilot also supports personal instructions via your IDE settings and organization-level instructions through Copilot settings.

**Copilot Memory** (preview) takes this further -- Copilot learns details about a repo over time and applies them automatically.

### Cross-tool compatibility

If your team uses multiple tools, start with a `CLAUDE.md` at the project root. OpenCode reads it as a fallback. For Copilot, maintain a separate `.github/copilot-instructions.md` -- the format expectations are different enough that a shared file gets awkward.

The goal isn't one file to rule them all. It's making sure every tool knows your project's conventions from the first prompt.

### Try it

Run `/init` in Claude Code or OpenCode. Review what it generates. Then add one thing it couldn't discover -- a team convention, a deployment quirk, an architectural decision. That's the stuff that makes configuration valuable.
