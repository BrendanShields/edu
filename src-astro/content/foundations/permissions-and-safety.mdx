# Permissions & Safety

Imagine giving someone access to your terminal and saying "fix everything." They could read any file, run any command, delete anything, push to production. You'd want some ground rules first.

That's exactly the problem permission systems solve. Every AI coding tool can read files, edit code, run shell commands, and interact with external services. The question is: how much should it do without asking?

### The permission spectrum

Think of it as a dial. On one end, the agent asks before every action. On the other, it does whatever it thinks is right. Both extremes are bad.

Too restrictive? You spend all your time clicking "approve" on obvious actions. The agent reads a file -- approve. It runs `git status` -- approve. You become a human rubber stamp.

Too permissive? The agent runs a database migration you weren't ready for. Or `rm -rf`s a directory it thought was temporary. Or pushes untested code to main.

The sweet spot depends on what you're doing right now. Exploring unfamiliar code? Be restrictive. Running tests on a project you know well? Loosen up.

### Claude Code: six modes

Claude Code has the most granular permission system. Six distinct modes, each a different tradeoff.

**Plan mode** is read-only. The agent explores your code and produces a plan, but can't change anything. Use this when you're investigating.

**Default mode** lets the agent read freely but asks before edits or shell commands. This is where most people start.

**Accept Edits mode** allows reads and edits without asking, but still prompts for shell commands. Good for active development when you're watching the output.

**Auto mode** uses a background safety classifier to evaluate every action. No prompts unless something looks risky. Available on Team plans with recent Claude models.

**DontAsk mode** only allows actions matching your explicit allow rules. Everything else is silently denied. Built for CI pipelines and scripts.

**Bypass mode** disables all safety checks entirely. Every action executes immediately. Only use this in isolated containers with no internet access.

You can press `Shift+Tab` to cycle modes during a session. And beyond modes, you can set granular allow/ask/deny rules for specific tools and command patterns:

```json
{
  "permissions": {
    "allow": [
      "Bash(npm test)",
      "Bash(git diff *)",
      "Read"
    ]
  }
}
```

This lets you say "always allow `npm test` without asking, even in default mode."

Every file edit creates a checkpoint. If something goes wrong, press `Esc` twice to rewind. Checkpoints are separate from git and local to your session.

### OpenCode: allow, ask, deny

OpenCode takes a simpler approach. Each tool gets one of three states: `allow`, `ask`, or `deny`.

By default, everything is allowed. That's the most permissive starting point. To tighten things up, configure permissions in `opencode.json`:

```json
{
  "permission": {
    "bash": {
      "*": "ask",
      "git *": "allow",
      "npm *": "allow",
      "rm *": "deny"
    },
    "edit": "ask"
  }
}
```

The last matching rule wins. Put broad catch-all rules first, specific overrides after.

OpenCode also has agent-level permissions. The Plan agent defaults to `ask` for everything, preventing accidental modifications during analysis. And when prompted for approval, you can approve once, approve always (for the session), or reject.

Use `/undo` to revert changes and `/redo` to restore them.

### GitHub Copilot: the IDE as safety net

Copilot's permission model is fundamentally different because it lives in your editor.

For inline suggestions, the model is dead simple: Tab to accept, Esc to dismiss. No code enters your file without a keystroke.

In Chat and Edit mode, changes appear as diffs. You review and apply each one individually.

Agent mode gives Copilot more autonomy -- it can decide which files to edit and propose terminal commands -- but commands still require approval. You see diffs before they land.

The coding agent is the most autonomous. It works on GitHub issues independently, creating branches and PRs. But the safety net is the PR review itself. You review the agent's work before merging, just as you would for a human colleague.

### The practical rule

Start restrictive. Loosen as you build trust. Commit before starting major agentic work -- git is your ultimate undo.

And remember: file edits are generally reversible. Shell commands with external side effects -- database migrations, deployments, API calls -- are not. Every tool treats shell commands with more caution than file edits, and so should you.
