# Writing Effective Prompts

Here's a bad prompt:

```text
fix the login bug
```

The agent doesn't know which login bug. It doesn't know where to look. It doesn't know how to tell if it worked. So it guesses -- reads random files, makes changes that seem plausible, and gives you something that looks right but probably isn't.

Now here's a good one:

```text
Users report login fails after session timeout.
Check the auth flow in src/auth/, especially token refresh.
Write a failing test that reproduces the issue, then fix it.
```

Same task. Dramatically different results. What changed?

### The first element: context

The agent needs to know where to look. Point it at files, directories, or patterns. "Check the auth flow in `src/auth/`" saves the agent from searching your entire codebase.

In Claude Code and OpenCode, use `@` to reference files directly. In Copilot, use `@workspace` to scope to your project or `#file` to point at something specific.

### The second element: intent

State what you actually want. "Fix the login bug" is intent, but it's vague. "Reproduce the session timeout failure and fix the root cause" is intent with teeth. The more specific your goal, the less the agent has to guess.

### The third element: constraints

Constraints prevent the agent from going off the rails. "Don't add new dependencies." "Follow the existing pattern in `orders.ts`." "Keep the public API unchanged." Without constraints, the agent will find a solution -- but it might be one you'd never accept.

### The fourth element: verification

This is the one most people skip, and it's the most powerful. Tell the agent how to check its own work. "Run the tests after." "Verify the build still passes." "Take a screenshot and compare."

When you include verification, the agent can iterate autonomously. Implement, check, fix, check again. Without it, you're the only feedback loop.

### You don't always need all four

A typo fix? Just intent: "Fix the typo in the README header." A complex refactor? You probably want all four elements working together.

The pattern scales with complexity. Small tasks need less structure. Big tasks need more. But if you ever find yourself unhappy with the agent's output, check which element you left out. That's usually where things went wrong.
