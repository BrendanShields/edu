# Verification-Driven Development

The single highest-leverage thing you can do is give the AI something to check against.

Without verification, the agent writes code that looks plausible and hands it to you. You become the only feedback loop. You read every line, run it yourself, find the bugs, explain them back. The agent is generating. You're verifying.

With verification, everything flips. The agent implements, runs the tests, reads the failures, fixes the issues, runs the tests again. It iterates autonomously until the work is actually correct. You go from line-by-line reviewer to final approver.

That's the dramatic difference.

### Tests first, then code

The most reliable pattern is writing tests before implementation.

```text
Write tests for a RateLimiter class:
- Allows 10 requests per minute per user
- Returns 429 after limit exceeded
- Resets after the window expires
- Handles concurrent requests safely

Run the tests (they should fail), then implement
RateLimiter to make them pass.
```

This works because the tests define exact expected behavior. The agent has a clear success signal -- green tests. And it can iterate without needing you at all.

Compare that to "implement a rate limiter." The agent writes something. Does it work? Who knows. You'd have to read every line and check the logic manually.

### The verification spectrum

Tests aren't the only verification tool. Match the verification to the task.

For **bug fixes**: paste the error message. Tell the agent to fix it and verify the error is gone. "The build fails with `TypeError: Cannot read property 'id' of undefined` in `src/api/users.ts`. Fix it and verify the build succeeds."

For **refactoring**: tell the agent to run existing tests after each change. "Refactor `utils.js` to ES modules. Run existing tests after each file change. No test should break."

For **UI work**: paste a screenshot or mockup. Ask the agent to implement it, take a screenshot, compare, and fix differences. Visual verification closes the loop the same way tests do.

For **linting**: add your linter to the loop. "After each file change, run `npx tsc --noEmit` and fix any new errors before moving on." In Claude Code, you can set up hooks to auto-run linters after every file edit.

### Stacking verification layers

The best prompts combine multiple layers:

```text
Add a /users/:id endpoint.
- Follow the pattern in src/api/handlers/orders.ts
- Write tests first, then implement
- Run npm test after implementation
- Run npm run lint and fix any issues
- Run npm run typecheck and fix any issues
```

Tests catch logic errors. Linters catch style issues. Type checkers catch interface mismatches. Each layer catches a different class of problem.

### The compound effect

Without verification, you can delegate simple tasks. With verification, you can delegate complex ones. The agent doesn't just write code -- it proves the code works.

That's the difference between an assistant that generates plausible text and a collaborator that delivers working software.
