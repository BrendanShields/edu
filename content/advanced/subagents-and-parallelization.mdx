You need to understand a module before changing it. You ask the agent to explore it. Now your main conversation is fifty files heavier. The context is polluted with research you needed once, and every subsequent response is slightly worse because the agent is wading through noise.

There's a cleaner way to handle this.

### The Subagent Solution

A subagent is an isolated AI assistant with its own context window. The main agent sends it a task, it works independently, and returns a summary. Your main conversation never sees the intermediate steps -- just the conclusion.

Think of it like sending someone to the library. They do the research, bring back notes, and you keep working. Your desk stays clean.

### Built-In Subagents

Claude Code ships with three. **Explore** is fast and read-only, powered by Haiku -- great for searching codebases and answering questions about how things work. **Plan** handles research for planning mode. **General-purpose** has full tool access for complex multi-step tasks.

Claude decides when to delegate automatically. It even specifies thoroughness for Explore -- quick, medium, or very thorough -- based on the question's complexity.

OpenCode has a similar split. **General** handles multi-step tasks with full tool access. **Explore** is fast, read-only search. Invoke them manually with `@general` or `@explore`, or let the primary agent delegate on its own.

### Custom Subagents

This is where it gets interesting. Define a subagent in `.claude/agents/` (Claude Code) or `.opencode/agents/` (OpenCode):

```markdown
---
name: security-reviewer
description: Reviews code for security vulnerabilities
tools: Read, Grep, Glob
model: sonnet
---

Analyze code for SQL injection, XSS, hardcoded secrets,
and missing input validation. Rate each finding by severity.
```

Now you have a security specialist on call. Give it read-only tools so it can't accidentally change anything. Choose a faster model if speed matters more than depth. What specialized reviewer would help your team the most?

### Parallel Sessions With Git Worktrees

Subagents share the same repository. For truly parallel work -- where two agents might edit the same file -- use git worktrees. Each worktree is an independent working copy. Start a separate agent in each one, and multiple features progress simultaneously without file conflicts.

The `/batch` skill automates this pattern. It decomposes a large task into units, spawns one agent per worktree, and each opens its own PR.

### When to Delegate

Delegate when the task is exploratory, independent of conversation context, or parallelizable. A codebase search, a security review, a documentation sweep -- perfect for subagents.

Keep work inline when you need back-and-forth discussion or when the result immediately shapes your next prompt. Subagents don't see your conversation history, so they struggle with tasks that depend on recent context.

### Notes

- Subagents isolate research from your main conversation, preventing context pollution
- Claude Code has Explore (read-only, fast), Plan, and General-purpose subagents
- OpenCode has General (full access) and Explore (read-only) subagents
- Custom subagents in `.claude/agents/` or `.opencode/agents/` encode specialized roles
- Git worktrees enable true parallelism -- multiple agents, no file conflicts
- `/batch` automates the worktree pattern for large-scale changes
- Delegate exploratory work; keep interactive, context-dependent work inline
