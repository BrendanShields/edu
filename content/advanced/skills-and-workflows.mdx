You've explained your deployment process to the AI five times this week. Each time from scratch. "First check the staging environment. Then run the migration. Then deploy to canary. Then verify the health check. Then promote to production."

Five times. The exact same instructions. There's a better way.

### The Repetition Problem and the Skill Solution

Memory files load every session. Skills load on demand -- when you invoke them or when the agent decides they're relevant. A deployment skill doesn't sit in memory while you're debugging CSS. It shows up when you type `/deploy`.

Think of skills as recipes. The agent has the cooking ability. Skills give it the recipe for your specific dish.

### Creating a Skill

In Claude Code, skills live in `.claude/skills/`. Each is a directory with a `SKILL.md` file:

```yaml
---
name: fix-issue
description: Fix a GitHub issue by number
disable-model-invocation: true
---

Fix GitHub issue $ARGUMENTS following our coding standards.

1. Read the issue with `gh issue view $ARGUMENTS`
2. Search the codebase for relevant files
3. Implement the fix and write tests
4. Create a commit
```

Invoke it with `/fix-issue 42`. The `$ARGUMENTS` placeholder gets replaced with your input. Setting `disable-model-invocation: true` means only you can trigger it -- the agent won't decide to fix random issues on its own.

### Dynamic Context

Skills can inject live data before the prompt reaches the agent. Use the `` !`command` `` syntax:

```yaml
---
name: pr-summary
description: Summarize the current PR
context: fork
---

PR diff: !`gh pr diff`
Changed files: !`gh pr diff --name-only`

Summarize this pull request concisely.
```

Those shell commands run first. The agent sees the actual PR data, not the commands. This is preprocessing -- what workflow in your day-to-day would benefit from dynamic context injection?

### Cross-Tool Skills

OpenCode uses the same `SKILL.md` format and reads from `.claude/skills/` for compatibility. Copilot takes a different approach -- reusable prompts live in `.github/copilot/prompts/*.md` and you reference them in chat with `#prompt-name`. The concept is the same everywhere: package once, reuse forever.

### When to Create a Skill

Create a skill when you give the same multi-step instructions more than twice. Deployment procedures, PR reviews, issue triage, code generation from templates -- anything with a repeatable pattern. Keep `SKILL.md` focused. If it needs reference material, put it in supporting files within the skill directory.

### Notes

- Skills load on demand, unlike memory files which load every session
- Place skills in `.claude/skills/` with a `SKILL.md` file
- `$ARGUMENTS` lets you pass input; `disable-model-invocation` prevents agent self-triggering
- Dynamic context with `` !`command` `` injects live data before the prompt reaches the agent
- OpenCode reads `.claude/skills/` for compatibility; Copilot uses `.github/copilot/prompts/`
- Create a skill when you repeat the same multi-step instructions more than twice
