# Permissions & Safety

Imagine handing someone your laptop, logged into everything, and saying "fix whatever you want." Your email is open. Your terminal has admin access. Your production database credentials are in `.env`.

That's what happens when you give an AI coding tool full access without thinking about permissions.

### WHAT COULD GO WRONG

Three stories. All grounded. All preventable.

**The wrong directory.** You ask the agent to clean up some test fixtures. It runs `rm -rf` on what it thinks is a temp folder. It was `src/`. You didn't have a recent commit. Three hours of work, gone.

**The silent push.** The agent finishes a refactor and helpfully runs `git push origin main`. No tests. No review. Your CI pipeline deploys on push. Your users find out about the bug before you do.

**The table drop.** You're working on a migration. The agent generates a `DROP TABLE users` to recreate the schema from scratch. It runs the migration. In production. Your DBA's phone rings at 2am.

These aren't hypothetical. They're what happens without guardrails.

### THE PERMISSION SPECTRUM

Every tool has a dial. On one end: "ask me before you do anything." On the other: "do whatever you think is right."

Both extremes will ruin your day.

Too far left, and you're approving fifty prompts just to rename a variable. The agent reads a file — approve. It runs `git status` — approve. You become a human rubber stamp, and the entire point of using an agent evaporates.

Too far right, and you learn about the database migration after production is down. The agent made a reasonable decision. It just wasn't *your* decision.

The sweet spot isn't a fixed position. It moves based on what you're doing right now.

### FINDING YOUR LEVEL

I've found that trust with AI tools works exactly like trust with a new team member. You start cautious and loosen up as you learn what they'll do.

**Day 1 on a new codebase?** Plan mode. Read-only. Let the agent explore and explain, but don't let it touch anything. You're still learning the conventions yourself.

**Day 3, familiar with the patterns?** Default mode. The agent reads freely, but asks before edits and commands. You're reviewing each change, but you're not micromanaging.

**Week 3, you trust the conventions?** Auto mode. The agent handles routine work independently. You check in on the results, not every individual action.

Where are you right now? That's your starting mode.

And one rule that never changes: **commit before starting major agentic work.** Git is your ultimate undo button, and it works even when everything else fails.

### HOW EACH TOOL DOES IT

**Claude Code** has six modes, each a different point on the dial.

*Plan mode* is read-only — exploring and producing plans, no changes. *Default mode* reads freely but asks before edits and shell commands. *Accept Edits mode* allows file changes without asking but still prompts for commands. *Auto mode* uses a safety classifier to evaluate each action, only prompting when something looks risky. *DontAsk mode* only runs actions matching explicit allow rules, silently denying everything else — built for CI. *Bypass mode* disables all checks entirely — only use this in isolated containers with no internet.

Press `Shift+Tab` to cycle modes mid-session. You can also set granular rules for specific tools and commands:

```json
{
  "permissions": {
    "allow": [
      "Bash(npm test)",
      "Bash(git diff *)",
      "Read"
    ]
  }
}
```

Every file edit creates a checkpoint. Something go wrong? Press `Esc` twice to rewind. Checkpoints are separate from git and local to your session.

**OpenCode** takes a simpler approach: each tool gets one of three states — `allow`, `ask`, or `deny`. By default, everything is allowed. To tighten things up:

```json
{
  "permission": {
    "bash": {
      "*": "ask",
      "git *": "allow",
      "npm *": "allow",
      "rm *": "deny"
    },
    "edit": "ask"
  }
}
```

The last matching rule wins. Broad rules first, specific overrides after. The Plan agent defaults to `ask` for everything. Use `/undo` to revert and `/redo` to restore.

**GitHub Copilot** uses the IDE itself as the safety net. Inline suggestions require Tab to accept — no code enters your file without a keystroke. Chat and Edit mode shows changes as diffs you review individually. Agent mode can choose files and propose terminal commands, but commands still need approval. The coding agent works most independently — creating branches and PRs from issues — but the PR review is the guardrail.

Can you spot the pattern? More autonomy always comes with a different kind of review gate.

### NOTES

- Start restrictive. Build trust. Loosen gradually.
- Never bypass in production. Ever.
- File edits are reversible. Shell commands with side effects — migrations, deploys, API calls — are not.
- Every tool treats shell commands with more caution than file edits. So should you.
- Commit before you start. Git is the safety net underneath all the other safety nets.
