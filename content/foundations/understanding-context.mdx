# Understanding Context

If you learn one concept from this workshop, make it this one.

Every AI coding tool has a hard limit on how much it can hold in its head at once. This limit is called the context window. It determines everything — how well the agent understands your code, how accurately it follows your instructions, and when it starts silently falling apart.

### THE DESK

Think of the context window as a desk.

Everything the agent works with has to fit on this desk: your conversation, every file it's read, every command output, its own system instructions, even the definitions of its tools. All of it, physically spread across one surface.

A clean desk with the right documents lets you think brilliantly. A desk buried under every piece of paper from the entire project makes you lose things, miss connections, and do sloppy work.

What happens when the desk is full and you need to add something new?

Something old gets pushed off the edge. The agent starts forgetting — instructions you gave at the start, files it read earlier, decisions you made together. Not because it's broken. Because the desk is full.

### WHAT FILLS IT UP

Not everything costs the same. This is where people get into trouble.

Reading a small config file? Cheap. A short prompt and response? Cheap. A grep that returns a few matches? Cheap.

Reading a 2,000-line file? Expensive. Running a test suite and dumping the full output? Very expensive. Having twenty back-and-forth exchanges? The cost compounds with every message. Connecting multiple MCP servers that each add tool definitions to every request? Surprisingly expensive.

Here's rough scale. Claude Code gives you 200K tokens by default, up to 1M with extended context. OpenCode depends on your model — GPT-4o gets ~128K, Claude Sonnet ~200K, Gemini over 1M. Copilot varies by model and uses workspace indexing to stay efficient.

A token is roughly three-quarters of a word. So 200K sounds like a lot — until a single large file read eats 5,000 tokens and a verbose test output eats 10,000 more.

I've seen people fill half their context just by pasting error logs. One copy-paste, and half the desk is covered in a stack of paper you'll never look at again.

### THE MOMENT IT BREAKS

Here's the thing that makes context exhaustion dangerous: **it doesn't crash the tool. It degrades silently.**

First, the agent starts forgetting instructions you gave at the beginning of the conversation. You said "use single quotes for strings" in message two. By message thirty, it's back to double quotes. Not because it's ignoring you. Because that instruction slid off the desk.

Then it starts repeating work. It reads a file it already read. It tries an approach you already rejected. It asks you a question you already answered.

Eventually, responses get generic. The agent loses the thread of your specific problem and falls back to general patterns. It makes mistakes it wouldn't have made twenty minutes ago.

Have you ever noticed the agent repeating itself? Suggesting something you explicitly told it not to do? That's context exhaustion. And by the time you notice it, you've already wasted several exchanges trying to correct an agent that literally can't remember why it's wrong.

### HOW TO MANAGE IT

You manage context like currency. Spend wisely. Save often. Don't hoard.

**Start fresh between tasks.** Use `/clear` or `/new` when switching to a different problem. Investigation in one session. Implementation in a new session. Testing in a third. Each session starts with a clean desk. You might think carrying context forward helps — it almost never does.

**Compress when context is high.** Run `/compact` when you notice the agent slowing down or repeating itself. Add a focus to keep what matters: `/compact focus on the API changes`. This summarizes older exchanges and clears out raw tool outputs.

**Send research to subagents.** In Claude Code, subagents get their own fresh desk. They do the work — reading files, searching code, running commands — and return just a summary to your main session. OpenCode's `@explore` and `@general` work the same way. Think of it as sending an intern to the library instead of hauling every book to your desk.

**Put recurring instructions on the wall.** CLAUDE.md, AGENTS.md, `.github/copilot-instructions.md` — these are sticky notes on the wall above your desk. They load at the start of every session and survive compaction. Put your coding conventions, project structure, and preferences there instead of repeating them in every conversation. Keep them lean — under 200 lines. Every line takes space from your actual work.

The best sessions are the ones where you manage context deliberately. Not the ones where you dump everything in and hope for the best.

### NOTES

- Context is the #1 skill for working with AI coding tools. Everything else is secondary.
- Manage it like money: spend wisely, save often, don't hoard.
- When the agent starts forgetting or repeating — compact, or start fresh. Don't fight a full desk.
- Subagents are free desks. Use them for research instead of bloating your main session.
- Persistent instruction files are the wall above the desk. They survive everything.
