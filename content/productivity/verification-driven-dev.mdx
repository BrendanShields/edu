# Verification-Driven Development

### The Anxiety of 200 Lines

You just asked the AI to refactor your auth module. It changed 200 lines across 8 files. Now what?

You're staring at a diff you didn't write. You can't fully trace the logic. You don't have tests covering half of it. You read every line, squinting at edge cases, hoping you catch whatever might be wrong. This is the worst part of working with AI -- reviewing code on faith.

### The Same Refactor, Different Approach

Now imagine you'd said this instead: "Write a failing test for the current auth behavior first, then refactor."

The agent writes 20 lines of tests that capture exactly what the module should do. Tests fail (nothing's changed yet). Then the agent refactors. Tests pass. You review 20 lines of test assertions instead of 200 lines of implementation. You know *what* correctness means because you can read it.

Would you rather review 200 lines of code or 20 lines of tests?

### Tests First, Then Code

The most reliable pattern is writing tests before implementation.

```text
Write tests for a RateLimiter class:
- Allows 10 requests per minute per user
- Returns 429 after limit exceeded
- Resets after the window expires

Run the tests (they should fail), then implement
RateLimiter to make them pass.
```

The tests define exact expected behavior. The agent has a clear success signal -- green tests. It can iterate without needing you at all.

### The Verification Spectrum

Tests aren't the only way to close the loop. I've found that matching the verification to the task is what makes this work in practice.

**Bug fixes:** Paste the error message. "Fix it and verify the error is gone."

**Refactoring:** "Run existing tests after each change. No test should break."

**UI work:** Paste a screenshot or mockup. "Implement it, take a screenshot, compare, fix differences."

**Linting and types:** "After each file change, run `npx tsc --noEmit` and fix any new errors." In Claude Code, set up hooks to auto-run linters after every edit.

### Stacking the Layers

The best prompts combine multiple verification layers -- tests catch logic errors, linters catch style issues, type checkers catch interface mismatches. Each layer catches a different class of problem, and together they let the agent self-correct through nearly anything.

### Notes

- Without verification, you become the only feedback loop -- reading every line on faith
- Writing tests first lets you review 20 lines of assertions instead of 200 lines of implementation
- The verification spectrum: **tests** for logic, **linters** for style, **type checkers** for interfaces, **screenshots** for UI
- Stacking multiple verification layers lets the agent iterate autonomously until the work is actually correct
- Verification is what turns the agent from a code generator into a collaborator that delivers working software
