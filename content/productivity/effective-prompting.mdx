# Writing Effective Prompts

### The Prompt That Wasted Ten Minutes

```text
fix the login bug
```

Here's what the agent sees when it reads that: nothing. No direction, no scope, no way to know if it succeeded. So it guesses. It reads random files in `src/`, makes changes that seem plausible, and hands you something that looks right but fixes the wrong bug. Sound familiar?

### What the Agent Actually Sees

From the agent's perspective, a vague prompt is like being dropped into a foreign city with no map. It has to make assumptions about every single thing -- which files matter, what "fixed" means, whether you care about side effects. Each assumption is a coin flip.

A specific prompt is a GPS route. The agent knows where to look, what success means, and what to avoid. The difference isn't intelligence. It's information.

### Context: Where to Look

```text
Check the auth flow in src/auth/, especially token refresh.
```

You've just saved the agent from searching your entire codebase. In Claude Code and OpenCode, use `@` to reference files directly. In Copilot, use `@workspace` or `#file` to scope things down.

### Intent: What You Actually Want

"Fix the login bug" is intent, but it's vague. "Reproduce the session timeout failure and fix the root cause" is intent with teeth. I've found that the more specific your goal, the less the agent invents on its own.

### Constraints: What to Avoid

"Don't add new dependencies." "Follow the existing pattern in `orders.ts`." "Keep the public API unchanged." Without constraints, the agent will find *a* solution -- but it might be one you'd never accept.

### Verification: How to Prove It Worked

This is the one most people skip, and it's the most powerful. "Run the tests after." "Verify the build still passes." When you include verification, the agent can iterate on its own -- implement, check, fix, check again. Without it, you're the only feedback loop.

Which of these four elements do you usually skip?

You don't always need all four. A typo fix just needs intent. A complex refactor needs everything. But when you're unhappy with the agent's output, check which element you left out. That's usually where things went wrong.

### Notes

- A vague prompt forces the agent to guess at every step -- context, intent, constraints, and success criteria
- The four elements are **context** (where to look), **intent** (what you want), **constraints** (what to avoid), and **verification** (how to prove it worked)
- Verification is the highest-leverage element because it lets the agent self-correct without you
- Scale the structure to the task: small tasks need less, complex tasks need all four
